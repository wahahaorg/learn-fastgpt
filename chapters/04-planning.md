# 04 任务拆解与计划

## 本章目标

复杂任务需要计划。没有计划的 Agent 容易出现三个问题：乱调工具、忘记目标、过早给出答案。

本章会讲：

- 为什么需要计划。
- 计划应该长什么样。
- 如何让模型更新计划。
- 什么时候允许 Agent 停止。

## 为什么需要计划

用户常常给的是目标，不是步骤：

```txt
帮我分析这份用户反馈，找出最重要的问题，并写一份改进建议。
```

这个目标至少包含：

1. 读取反馈。
2. 清洗和分类。
3. 统计高频问题。
4. 判断优先级。
5. 生成建议。
6. 输出报告。

如果没有计划，模型可能直接写一份泛泛而谈的建议。计划让 Agent 在行动前先形成可检查的路线。

## 计划的数据结构

计划不要只是一段自然语言。建议结构化：

```ts
type PlanStep = {
  id: string;
  title: string;
  status: 'pending' | 'in_progress' | 'done' | 'blocked';
  evidence?: string;
};

type Plan = {
  goal: string;
  steps: PlanStep[];
};
```

每一步都应该能被验证。比如“分析数据”太宽泛，“统计每类反馈数量”更可执行。

## 计划不是越细越好

太粗的问题是不可执行，太细的问题是成本高。

不好的计划：

```txt
1. 完成任务。
```

过细的计划：

```txt
1. 打开文件。
2. 读取第一行。
3. 读取第二行。
4. 读取第三行。
```

合适的计划：

```txt
1. 读取反馈数据并识别字段。
2. 按主题聚类反馈。
3. 统计每类反馈的数量和代表样例。
4. 生成优先级排序。
5. 输出改进建议。
```

## 计划生成

可以让模型先生成计划：

```txt
请先给出解决该任务的步骤。
要求：
- 每一步必须可验证。
- 如果缺少信息，标记 blocked。
- 不要开始执行工具。
```

这一步把“想”和“做”分开。对复杂任务很有用。

## 计划更新

执行过程中，计划会变化：

- 工具失败，需要换方法。
- 用户补充了新目标。
- 发现原计划不够。
- 某一步已经完成。

因此 Agent 需要一个内部工具，比如 `update_plan`：

```ts
type UpdatePlanInput = {
  stepId: string;
  status: PlanStep['status'];
  evidence?: string;
};
```

模型调用它时，不是改变外部世界，而是改变自己的任务状态。

## 停止条件

Agent 什么时候可以输出最终答案？

至少满足：

- 必要步骤已完成。
- 没有未处理的 blocker。
- 工具结果足够支持结论。
- 没有明显缺失的用户输入。
- 没有超过安全边界。

可以实现一个 stop gate：

```ts
function canStop(plan: Plan): boolean {
  return plan.steps.every((step) => step.status === 'done');
}
```

真实系统会更复杂，但最小版本先这样就够。

## 什么时候不需要计划

不是所有任务都要计划。简单问答如果强制计划，会让体验变差。

可以按任务复杂度判断：

- 一步能回答：不需要计划。
- 需要一个工具：可以不显式计划。
- 需要多个工具或多轮推理：需要计划。
- 需要用户确认或高风险操作：必须计划。

## 追问用户

计划中如果出现 blocker，不要让模型猜。

例如：

```txt
目标：帮我给客户发邮件。
缺失：客户邮箱、邮件语气、是否需要附件。
```

此时 Agent 应该询问用户：

```txt
我需要确认客户邮箱和邮件语气。请提供邮箱，并选择正式、友好或简洁风格。
```

追问本质上也是一种工具：它暂停当前任务，等待用户补充信息。

## 本章练习

给你的 Agent 增加计划能力：

1. 如果用户任务超过一句话，先生成计划。
2. 执行工具前标记当前步骤为 `in_progress`。
3. 工具成功后标记为 `done`，并保存 evidence。
4. 如果缺少信息，返回追问。
5. 只有所有必要步骤完成后才输出最终答案。

做完这一步，你的 Agent 会更慢一点，但可控性会明显提升。
